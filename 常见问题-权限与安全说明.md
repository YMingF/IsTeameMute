# 常见问题：权限与安全说明

本文解释 Teams Mic Control 安装或启动后，为什么可能会看到 macOS 的权限提示，以及这些权限对电脑安全的实际影响。

## 1. 为什么启动后会弹出“密钥”相关提示？

这里的“密钥”通常指的是 macOS 的 `钥匙串（Keychain）` 访问提示，不是应用在索取你的系统主密码，也不是要访问你所有账号密码。

这个项目会把 Teams 本地 API 配对成功后返回的 token 保存到 macOS 钥匙串，后续启动时再读取它，避免每次启动都重新配对。

相关实现位置：

- [Sources/TeamsMuteOverlayCore/KeychainTokenStore.swift](./Sources/TeamsMuteOverlayCore/KeychainTokenStore.swift)
- [Sources/TeamsMuteOverlayCore/TeamsOverlayController.swift](./Sources/TeamsMuteOverlayCore/TeamsOverlayController.swift)
- [README.md](./README.md)

代码里使用的默认 Keychain 标识是：

- `service`: `TeamsMuteOverlay`
- `account`: `teams-api-token`

因此，如果你看到类似下面含义的弹窗：

```text
TeamsMuteOverlay 想要访问钥匙串中的 “TeamsMuteOverlay”
```

它通常是在读取或写入这条 Teams 配对 token，而不是在请求更高系统权限。

## 2. 为什么它需要这个 token？

这个应用依赖 Microsoft Teams 桌面端暴露在本机的 Third-party app API。

首次配对时，Teams 会弹出 `Allow / Block` 提示。用户允许后，Teams 会返回一个 token。应用把这个 token 存入 Keychain，后续再用它重新连接本地 Teams API，完成以下功能：

- 获取会议状态
- 获取 Teams 当前静音状态
- 在 API 允许时执行静音切换
- 支持“Teams 会议开始时自动启动”的 helper 进程复用同一个 token

如果没有这个 token，应用通常需要重新进入配对流程。

## 3. 这是否意味着它能访问我电脑里的所有密码？

不是。

从当前项目代码看，它只实现了对自己那条 Keychain 记录的读写逻辑，用来保存 Teams API token。没有看到读取浏览器密码、系统账号密码、Wi-Fi 密码或其他通用凭据的逻辑。

但需要区分两件事：

1. **当前代码用途**  
   这里只是保存和读取 `Teams API token`。
2. **权限本质**  
   一旦某个应用被允许访问某条 Keychain 项，它就可以继续访问那条它获准使用的记录。

所以，这个权限的实际风险点不是“它能拿到你所有密码”，而是“它能拿到它自己保存在 Keychain 里的 Teams 配对 token”。

## 4. 它还会请求哪些权限？

当前仓库里还能确认到一个明确权限：`麦克风权限`。

相关实现位置：

- [Sources/TeamsMuteOverlayCore/MicrophoneLevelMonitor.swift](./Sources/TeamsMuteOverlayCore/MicrophoneLevelMonitor.swift)
- [dist-20260512-170748/TeamsMuteOverlay.app/Contents/Info.plist](./dist-20260512-170748/TeamsMuteOverlay.app/Contents/Info.plist)
- [dist-20260512-170748/TeamsMuteOverlay.entitlements](./dist-20260512-170748/TeamsMuteOverlay.entitlements)

用途是：

- `音量 Pulse`
- `静音讲话警告`
- 悬浮按钮的实时音量可视化

文档和代码都表明，这部分能力是把麦克风输入当作“实时电平计”使用，而不是录音保存。

## 5. 麦克风权限会带来什么安全影响？

从当前代码实现看：

- 应用通过 `AVAudioEngine` 读取输入流
- 只计算实时峰值 `level`
- 没有看到把音频写入磁盘或上传网络的逻辑

这说明它当前的设计目标是“本地实时音量检测”，不是录音。

但也要实话实说：

只要一个应用拿到了麦克风权限，从系统能力上讲，它就具备读取音频输入的可能性。因此，麦克风权限本身始终属于敏感权限，即使当前代码只用了其中很小的一部分能力。

## 6. 它会不会在后台长期运行？

项目里包含一个 login item helper，用于支持菜单项 `Teams 会议开始时自动启动`。

相关位置：

- [Sources/TeamsMuteOverlayMeetingHelper/main.swift](./Sources/TeamsMuteOverlayMeetingHelper/main.swift)
- [Sources/TeamsMuteOverlayApp/MeetingAutoLaunchService.swift](./Sources/TeamsMuteOverlayApp/MeetingAutoLaunchService.swift)

它的作用是：

- 登录后静默运行
- 等待 Teams API 报告 `isInMeeting == true`
- 然后启动主应用

这不等于获取了更高系统权限，但意味着如果你开启了这个选项，应用相关进程可以在登录后自动运行。

## 7. 从当前代码看，这个应用的权限风险大吗？

按当前仓库可见代码判断，权限风险主要分成三类：

### A. Keychain token 访问

用途明确，风险中等偏低。

- 优点：把 token 放 Keychain 比明文写配置文件更安全
- 风险：应用可访问自己保存的 Teams token

### B. 麦克风访问

用途明确，但权限敏感度高于 Keychain。

- 优点：只用于本地实时电平检测
- 风险：麦克风权限天然敏感

### C. 自动启动 helper

权限级别不高，但会影响进程运行方式。

- 优点：可自动响应 Teams 会议开始
- 风险：登录后自动运行，增加后台常驻行为

## 8. 从当前仓库看，没有看到哪些高风险权限？

截至当前代码与打包产物，未看到应用主动申请以下常见高风险权限：

- 辅助功能（Accessibility）
- 输入监控（Input Monitoring）
- 全盘访问（Full Disk Access）
- 屏幕录制（Screen Recording）

如果你启动时看到的是这些权限中的任何一种提示，就需要重新核对当前安装包是否就是本仓库构建出来的版本，或者是否引入了后续未审查的改动。

## 9. 如果我想更谨慎地使用它，应该怎么做？

可以按下面方式控制风险：

1. 首次运行时，仔细看 macOS 弹窗的原文，不要只看“允许”按钮。
2. 如果你不需要 `音量 Pulse` 或 `静音讲话警告`，可以拒绝麦克风权限。
3. 如果你不需要自动随会议启动，不要开启 `Teams 会议开始时自动启动`。
4. 如果不再使用，执行菜单里的“断开并清除 token”，删除保存的 Teams 配对 token。
5. 在 macOS 的“系统设置 -> 隐私与安全性”里检查并撤销不需要的授权。

## 10. 一句话结论

如果你看到的是 `钥匙串 / Keychain` 提示，它通常是在保存或读取 Teams API 配对 token，不是在索取你的系统总密码。  
如果你看到的是 `麦克风` 提示，它是为了本地实时音量检测。  
从当前仓库代码看，没有发现它会录音落盘、读取全盘文件、请求辅助功能或输入监控等更高风险权限。
